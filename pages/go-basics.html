Go Fundamentals

Everything a senior backend engineer must internalize before touching channels: types, slices, maps, structs, methods, interfaces, errors, and modules.

Mental model

  • Statically typed, compiled, garbage-collected language with first-class concurrency. The compiler is fast, the runtime is small, the binary is single.
  • Composition over inheritance. No classes, no inheritance — interfaces are satisfied implicitly. struct + methods + interface is the whole object system.
  • Errors are values, not exceptions. panic is for programmer bugs only, never for control flow.
  • Zero values are useful. A freshly-declared struct, slice, or map is a working empty thing, not nil-pointer minefield (with two exceptions: maps and channels need make for writes).
  • The standard library is the framework. net/http, encoding/json, database/sql, context, testing, log/slog are first-rate; reach for third-party only when you need extras.

1. Modules & project layout

// go.mod is the unit of versioning. Always pin Go and module path.
module github.com/acme/billing

go 1.22

require (
    github.com/google/uuid v1.6.0
    go.uber.org/zap v1.27.0
)
  • Module path == import path. Public packages live under that path; internal/ is enforced by the compiler.
  • Canonical layout: cmd/<app>/main.go, internal/... for private, pkg/... only if you publish reusable libs.
  • go mod tidy after every dependency change. go vet ./... + staticcheck in CI.

2. Variables, constants, iota

var x int            // zero value 0
y := 42              // short form, function scope only
const Pi = 3.14159   // untyped constant – no allocation, fits any numeric type

type Status int
const (
    StatusUnknown Status = iota // 0
    StatusPending               // 1
    StatusActive                // 2
)

// Bit flags
const (
    Read  = 1 << iota // 1
    Write             // 2
    Admin             // 4
)

3. Strings, runes, bytes

  • A string is an immutable read-only slice of bytes (UTF-8 encoded).
  • len(s) = bytes, not code points. Iterate with for i, r := range s to get runes.
  • Heavy concatenation → use strings.Builder (amortized O(n)).
var b strings.Builder
b.Grow(1024)
for _, p := range parts {
    b.WriteString(p)
}
result := b.String()

4. Arrays vs slices

// Allocate with known capacity to avoid repeated growth.
buf := make([]int, 0, 1024)
for i := 0; i < 1000; i++ {
    buf = append(buf, i)
}

// Common bug: slicing exposes shared array
a := []int{1, 2, 3, 4, 5}
b := a[:2]
b = append(b, 99)  // may overwrite a[2] if cap > len!

// Safe copy
c := append([]int(nil), a...)

5. Maps

m := make(map[string]int, 64)   // pre-sized hint
m["k"] = 1
v, ok := m["k"]                  // comma-ok
delete(m, "k")
for k, v := range m { _, _ = k, v } // iteration order is randomized
Reading from a nil map is OK (zero value), writing panics. Don't take pointers to map values – they're not addressable.

6. Structs, methods, embedding

type User struct {
    ID   uint64
    Name string
}

func (u User) Display() string  { return u.Name } // value receiver
func (u *User) Rename(n string) { u.Name = n }    // pointer receiver

// Embedding (composition, not inheritance)
type Auditable struct{ CreatedAt time.Time }
type Account struct {
    Auditable          // promoted: a.CreatedAt
    Balance int64
}
Rule of thumb: pointer receivers if the method mutates state OR the struct is large OR you want a single method set; otherwise value receivers. Stay consistent within a type.

7. Interfaces — the killer feature

type Reader interface {
    Read(p []byte) (n int, err error)
}

// Implicit satisfaction: any type with the right methods is a Reader.
type StringReader struct{ s string; i int }
func (r *StringReader) Read(p []byte) (int, error) {
    if r.i >= len(r.s) { return 0, io.EOF }
    n := copy(p, r.s[r.i:])
    r.i += n
    return n, nil
}
  • Define interfaces on the consumer side, not the producer side. Keep them small (often 1–2 methods).
  • Interface value = (type descriptor, value). nil interface ≠ typed nil:
func badNil() error {
    var e *MyError = nil
    return e          // returns a non-nil interface wrapping a nil *MyError
}
// Caller's `if err != nil` will be true. Always `return nil` explicitly.

8. Errors

// Sentinel
var ErrNotFound = errors.New("not found")

// Custom typed error
type ValidationError struct{ Field, Msg string }
func (e *ValidationError) Error() string { return e.Field + ": " + e.Msg }

// Wrapping & inspection
if err := repo.Get(id); err != nil {
    return fmt.Errorf("get user %d: %w", id, err)
}
if errors.Is(err, ErrNotFound)       { /* sentinel match */ }
var ve *ValidationError
if errors.As(err, &ve)               { /* typed match */ }
Never panic across API boundaries. panic is for programmer errors; recover only at goroutine roots (e.g. HTTP middleware).

9. Defer, panic, recover

func read(path string) (data []byte, err error) {
    f, err := os.Open(path)
    if err != nil { return nil, err }
    defer f.Close() // LIFO; runs even on panic

    return io.ReadAll(f)
}

// Capture panics at goroutine boundaries.
func safeGo(fn func()) {
    defer func() {
        if r := recover(); r != nil {
            log.Printf("panic: %v\n%s", r, debug.Stack())
        }
    }()
    fn()
}
  • Deferred calls evaluate arguments at defer time, body at return time.
  • Deferred function reading named returns can mutate them — useful for instrumentation.

10. Generics (1.18+)

type Number interface { ~int | ~int64 | ~float64 }

func Sum[T Number](xs []T) T {
    var s T
    for _, x := range xs { s += x }
    return s
}

// Type sets, not just unions:
type Ordered interface {
    ~int | ~int64 | ~float64 | ~string
}
func Max[T Ordered](a, b T) T { if a > b { return a }; return b }
Don't reach for generics first. They're for containers and algorithms over many types — not for replacing interfaces or for service plumbing.

Senior pitfalls

  • Returning typed-nil errors → tests pass, prod blows up.
  • Holding onto a slice of a huge buffer keeps the whole array alive (memory leak). slices.Clone when in doubt.
  • Closing a channel in a producer/consumer fan-in without coordination → panic.
  • Range-loop variable capture pre-1.22: for _, v := range xs { go f(v) } all see same v. Fixed in Go 1.22, still common in older codebases — copy explicitly.
  • Empty interface (any) is the new interface{}; prefer typed parameters or generics.
  • Using a pointer receiver on a small immutable struct — extra heap pressure, no upside.
  • append's aliasing trap — appending to a sub-slice may stomp on the original. Defensive append([]T(nil), s...) when handing off.
  • Mixing value and pointer receivers on the same type — half the methods satisfy the interface, the other half don't. Stay consistent.

Anti-patterns — what NOT to say

  • "I'll define an interface for every struct, just in case." Define interfaces on the consumer side; over-abstraction is a smell, not a virtue.
  • "panic is fine for validation errors." It's not — it's for unrecoverable programmer errors. Return errors.
  • "Getters and setters everywhere." In Go, an exported field is the public API. Encapsulate when it earns its keep, not by reflex.
  • "init() for everything." init hides ordering bugs and breaks tests. Prefer explicit constructors.
  • Treating empty interfaces as a Java Object escape hatch. If you find yourself reaching for any, reach for generics or a sum-type-style interface instead.
  • "Just go it" — every go fn() needs a clear shutdown story.

Senior interviewer follow-ups

ProbeWhat they want to hear
"Why do map and chan need make, but slices don't always?"Maps and channels are reference types backed by runtime structs that need allocation. A nil slice is still a valid empty slice — append works on it.
"What does x := s[2:4] share with s?"Backing array. Mutating one mutates the other (within the shared range).
"How does Go's defer stack work?"LIFO, runs on every return path including panic. Args evaluated at defer time, body at unwind.
"Why is interface nil-comparison weird?"An interface value is (type, value). nil interface means both are nil. A typed-nil — (*T)(nil) wrapped in an interface — is non-nil.
"When do you reach for generics?"Containers, algorithms, helpers over many types. Almost never for service layers.
"How do constants differ from variables?"Constants are compile-time, untyped by default, infinitely-precise numerics, no addresses.
"What's the cost of method-value vs method-expression vs receiver call?"Direct call = static dispatch, free. Interface call = itab lookup. Method value = closure allocation if it escapes.

Self-check

  • I can explain slice header (ptr, len, cap) and when append reallocates.
  • I know why typed-nil errors return non-nil interfaces — and how to avoid it.
  • I can choose value vs pointer receivers and explain the rule.
  • I can write a small interface, satisfy it implicitly, and explain its (type, value) pair-shape.
  • I can wrap and inspect errors with %w, errors.Is, errors.As.
  • I can sketch the anatomy of a go.mod + canonical project layout.
  • I understand iota, bit-flag enums, and untyped vs typed constants.
  • I know defer's arg-evaluation timing and LIFO order.
  • I can reason about strings as immutable byte slices and iterate runes correctly.
  • I can write a one-line generic Map[A,B] / Filter[T] / Reduce[T,A].
