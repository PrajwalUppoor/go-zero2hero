Microservices in Go

The senior-backend playbook: how to ship Go services that are fast, observable, resilient, and that fit cleanly into a distributed system.

Mental model

  • Default to a modular monolith. Microservices add operational burden you must justify with team scale, deploy cadence, or hard fault isolation.
  • Three constants in every service: clear bounded context, robust observability (metrics + logs + traces), and a defined SLO with error budget.
  • Layered Go layout: cmd/ → internal/<domain>/{transport, service, repository, storage}. Domain depends on nothing.
  • The 12 cross-cutting concerns are non-negotiable: config, log, metrics, trace, health, shutdown, timeouts, idempotency, authn/z, rate-limit, circuit-break, schema evolution.
  • Choose communication style per case: gRPC for east-west, REST for public, async events for decoupling, never RPC-then-publish.

When microservices are the right answer

  • Independent deploys — different teams ship at different cadences.
  • Different scaling profiles — one path is CPU-bound, another is I/O-bound.
  • Fault isolation — a slow recommendation service shouldn't take down checkout.
  • Polyglot persistence — one component needs Postgres, another Cassandra.
If you don't have multiple teams, multiple deploy cadences, or hard isolation needs, a well-structured monolith ("modular monolith") is almost always the better choice. Microservices trade simplicity for organizational scalability — pay the cost only when you reap the benefit.

Service anatomy — the layered Go layout

// /cmd/orderd/main.go        ← thin entrypoint (flags, config, wiring)
// /internal/order/...         ← business logic (no HTTP, no SQL leaks)
//     domain.go               ← entities, value objects, errors
//     service.go              ← use cases (Place, Cancel, Get)
//     repository.go           ← interface (Save, FindByID)
// /internal/order/transport/  ← HTTP/gRPC handlers, DTOs
// /internal/order/storage/    ← Postgres impl of repository
// /internal/platform/...      ← logger, metrics, tracing, db pool, kafka
// /api/openapi.yaml or *.proto
  • Dependency rule: transport → service → repository. Domain depends on nothing.
  • Interfaces live where they are used, not where they are implemented.
  • No global state. Pass dependencies explicitly; use a small wiring function in main.

Production-ready HTTP server in Go

func run(ctx context.Context) error {
    cfg := loadConfig()
    log := newLogger(cfg)

    db, err := pgxpool.NewWithConfig(ctx, mustParse(cfg.DSN))
    if err != nil { return err }
    defer db.Close()

    svc := order.NewService(order.NewPGRepo(db), kafkaPub)
    h   := transport.NewHTTP(svc, log)

    srv := &http.Server{
        Addr:              cfg.Addr,
        Handler:           h,
        ReadHeaderTimeout: 5 * time.Second,
        ReadTimeout:       15 * time.Second,
        WriteTimeout:      30 * time.Second,
        IdleTimeout:       60 * time.Second,
        MaxHeaderBytes:    1 << 20,
    }

    g, ctx := errgroup.WithContext(ctx)
    g.Go(func() error { return srv.ListenAndServe() })
    g.Go(func() error {
        <-ctx.Done()
        sd, cancel := context.WithTimeout(context.Background(), 10*time.Second)
        defer cancel()
        return srv.Shutdown(sd)
    })
    return g.Wait()
}
Why these timeouts matter: the default http.Server has none. A single misbehaving client can hold a goroutine forever, eventually starving the process. Always set them.

The 12 cross-cutting concerns every Go service must handle

  1. Configuration — env vars + flags, validated on boot, fail fast.
  2. Logging — structured (slog/zap), one log per request with trace ID.
  3. Metrics — Prometheus: RED (Rate, Errors, Duration) + USE (Utilization, Saturation, Errors).
  4. Tracing — OpenTelemetry, propagate W3C traceparent header, span per outbound call.
  5. Health checks — /healthz liveness, /readyz readiness (deps reachable).
  6. Graceful shutdown — drain in-flight, close DB pools, flush traces, exit 0.
  7. Timeouts & retries — context-everywhere, exponential backoff with jitter.
  8. Idempotency — Idempotency-Key header on writes, dedup store with TTL.
  9. Authn/z — JWT/OIDC at edge, mTLS east-west, never roll your own crypto.
  10. Rate limiting — per-tenant token bucket; protect downstreams.
  11. Circuit breaking & bulkheads — fail fast on failing deps, isolate resource pools.
  12. Schema evolution — Protobuf with backward-compatible fields; never reuse a tag.

Resilience patterns — Go snippets

Timeout + retry with backoff

func callWithRetry(ctx context.Context, do func(context.Context) error) error {
    var err error
    for attempt := 0; attempt < 5; attempt++ {
        cctx, cancel := context.WithTimeout(ctx, 800*time.Millisecond)
        err = do(cctx)
        cancel()
        if err == nil || !isRetriable(err) { return err }
        // Exponential backoff with full jitter.
        sleep := time.Duration(rand.Int63n(int64(time.Second << attempt)))
        select {
        case <-time.After(sleep):
        case <-ctx.Done(): return ctx.Err()
        }
    }
    return err
}

Bulkhead via bounded semaphore

sem := make(chan struct{}, 32) // at most 32 concurrent calls to dep X
func call(ctx context.Context) error {
    select {
    case sem <- struct{}{}:
        defer func() { <-sem }()
    case <-ctx.Done():
        return ctx.Err()
    }
    return doExpensiveCall(ctx)
}

Inter-service communication patterns

  • Request/response — gRPC or REST. Synchronous; couples availability.
  • Async events — Kafka/NATS. Decouples, but adds eventual consistency.
  • Saga — sequence of local txns + compensations for distributed workflows.
  • Outbox — atomic DB write + event publish; relay polls outbox table.
  • BFF (backend for frontend) — per-client aggregation; avoids chatty mobile/web calls.
  • API gateway — auth, rate-limit, routing, request shaping at the edge.

Configuration & secrets

  • 12-factor config — env vars; never check secrets into git.
  • Hot-reload sparingly — most things should require a restart for predictability.
  • Secrets — Vault, AWS Secrets Manager, Kubernetes Secrets + KMS.
  • Feature flags — separate from config; fast rollout/rollback without deploy.

Testing strategy (the Go pyramid)

  • Unit — pure functions, table-driven; mock external deps via interfaces.
  • Integration — real DB/broker via testcontainers-go; one happy path per repo.
  • Contract — gRPC/Pact tests so producer/consumer don't drift.
  • End-to-end — few; smoke test critical paths in staging.
  • Race & benchmarks — go test -race in CI; go test -bench for hot paths.

Deployment & operations

  • Containers — multi-stage Dockerfile, distroless or scratch base, non-root user, static binary.
  • Kubernetes — readiness probe, PodDisruptionBudget, HPA on RPS or CPU, requests/limits set.
  • Rollouts — blue/green or canary with metric gates (error rate, p99 latency).
  • SLOs — define availability/latency budgets; alert on burn rate, not raw thresholds.
  • Runbooks — every alert links to a doc with diagnosis steps.

Common Go pitfalls in production services

  • Forgetting defer rows.Close() — leaks DB connections under load.
  • Unbounded goroutines per request — backpressure with worker pools or semaphores.
  • Blocking on channels without ctx — every blocking op must select on ctx.Done().
  • Shared http.Client with no timeout — set Timeout AND tune Transport (idle conns, max per host).
  • Logging in hot paths at Info level — burns CPU and disk; gate behind a level check.
  • Returning maps as DTOs — define explicit structs; you'll thank yourself later.
  • Panicking in handlers — install a recovery middleware that logs + 500.

Senior-bar follow-ups interviewers ask

  • "How do you roll out a breaking schema change across services?" — expand-contract: add new field/route, dual-write, migrate consumers, remove old.
  • "How do you debug a tail-latency spike?" — RED dashboards → traces of slow requests → flamegraph from pprof → fix or shed.
  • "How do you handle a flapping dependency?" — circuit breaker, fallback (cached/default), alert, talk to the owning team.
  • "How do you guarantee at-least-once between two services?" — outbox + idempotent consumer, never RPC-then-publish.
  • "How do you make a Go service start cold < 1 s?" — small static binary, lazy-init non-critical deps, don't preload caches at boot.
