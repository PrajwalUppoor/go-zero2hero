Databases & Storage

Pick the right store; design the right schema; understand the engine. Senior interviews probe deeply here.

Mental model

  • Pick the storage primitive first: relational (rows + ACID), document (JSON), wide-column (partition + clustering), KV (sub-ms read), graph (traversals), search (inverted index), warehouse (column-store, OLAP).
  • Two storage engine families: B-tree (Postgres, MySQL InnoDB) for balanced read/write OLTP; LSM-tree (RocksDB, Cassandra) for high write throughput, sequential I/O.
  • Replicate for read scale + availability; shard for write scale + storage. They are different problems with different solutions.
  • Distributed transactions are slow and brittle. Saga + idempotency + outbox is the modern playbook; 2PC only when truly required.
  • "Index before you scale" — the right indexes turn O(n) scans into O(log n) lookups before you reach for sharding.

Storage engines

  • B-tree (Postgres, MySQL InnoDB) — balanced reads/writes, good for OLTP, in-place updates.
  • LSM-tree (Cassandra, RocksDB, LevelDB) — sequential writes, background compaction; high write throughput, read amplification.
  • Heap + indexes — Postgres; secondary indexes point to row TID.
  • Column-store (BigQuery, ClickHouse) — wide row scans, massive compression, OLAP queries.

  • Postgres default = Read Committed. MySQL InnoDB default = Repeatable Read (with gap locks).
  • For multi-statement workflows that must be atomic, wrap in a transaction; for high contention, prefer optimistic locking.

Indexing rules of thumb

  • Index every column that appears in WHERE, JOIN, ORDER BY for hot queries.
  • Composite index (a, b, c) serves queries on a, (a,b), (a,b,c) — not on b or c alone.
  • Covering index: include all selected columns to avoid a heap lookup.
  • Each index slows writes and grows storage; review usage with EXPLAIN / pg_stat_user_indexes.
  • Partial indexes for skewed predicates (WHERE status='pending').
  • Hash indexes for equality lookups; GIN/GiST for arrays, JSON, geo.

Sharding strategies

  • Hash sharding — even load, bad for range queries, hot keys still possible.
  • Range sharding — good for range queries, prone to hot ranges (e.g. by timestamp).
  • Directory-based — explicit lookup map per tenant; flexible but a SPOF risk.
  • Consistent hashing — minimal remap on resharding; use virtual nodes.

Replication

  • Single-leader (PG, MySQL) — strong on the leader, eventual on replicas; failover via Patroni/Orchestrator.
  • Multi-leader — multi-region writes; conflict resolution (LWW, CRDT, vector clocks).
  • Leaderless (Cassandra, Dynamo) — quorum R+W>N for strong consistency, read repair, hinted handoff.

Distributed transactions

  • Avoid two-phase commit across services — it's slow and partition-fragile.
  • Saga: sequence of local txns with compensating actions (choreography via events, or central orchestrator).
  • Transactional outbox: write business state + event row in same DB tx; relay publishes to broker.
  • Idempotency keys on every mutation API to allow safe retries.

Postgres-specific power moves

  • JSONB columns + GIN index for semi-structured data.
  • Logical replication for CDC into Kafka/ES.
  • Range partitioning by time + automated detach for retention.
  • SKIP LOCKED for queue-on-Postgres patterns.
  • SERIALIZABLE uses SSI — performance hit but no manual locks.

Cassandra-specific

  • Model around queries: pick a partition key that bounds rows per partition (~10s of MB).
  • Tunable consistency: ONE / QUORUM / ALL. R+W>N for strong.
  • No joins. Denormalize and write to multiple tables.

Backups & disaster recovery

  • Daily full + WAL-shipping incremental (RPO seconds, RTO minutes).
  • Point-in-time recovery is mandatory for finance/critical data.
  • Test restores periodically — an untested backup doesn't exist.
  • Cross-region replication; documented run-book for region failover.

Schema migrations safely

  • Expand-and-contract: add new column / table → backfill → switch reads → switch writes → drop old.
  • Avoid table rewrites in big tables (ALTER TABLE ... ADD COLUMN ... DEFAULT may rewrite); use partial backfills.
  • Online schema change tools (pt-online-schema-change, gh-ost) for MySQL.

Anti-patterns — what NOT to say

  • "SELECT * is fine." Returns more bytes, breaks covering indexes, leaks schema.
  • "Add an index on every column." Indexes cost write amplification + storage; review usage stats.
  • "We'll use eventual consistency." Where? On which reads? Spell it out.
  • Distributed 2PC across services. Almost always reach for Saga + outbox + idempotency.
  • Using a relational DB as a queue without SKIP LOCKED — head-of-line blocking.
  • JSON blobs as substitutes for proper columns when you'll query / index inside.
  • "Cassandra because scale." Without a query-shape that fits, Cassandra hurts.

Senior interviewer follow-ups

ProbeWhat they want to hear
"Why does your composite index help WHERE a=? AND b=? but not WHERE b=??"Indexes are sorted by leftmost column first; you need a prefix to use them.
"Snapshot isolation vs serializable — what's the gap?"Snapshot prevents dirty/unrepeatable/phantom but allows write-skew. Serializable closes write-skew at cost of more aborts.
"Saga vs 2PC?"2PC blocks on coordinator failure; saga uses compensating transactions and is partition-tolerant — at the cost of "compensable" semantics.
"What's a transactional outbox?"Write the business state and the event in the same DB transaction; a relay process publishes events from the outbox table.
"Postgres queue with SKIP LOCKED?"SELECT ... FOR UPDATE SKIP LOCKED LIMIT N — workers grab uncontested rows; no head-of-line blocking.
"Hot partition in Cassandra?"Bucket the partition key (user_id, day → user_id, day, bucket); cap rows per partition.
"Online schema change for a billion-row table?"Add nullable column → backfill in batches → add NOT NULL with default → switch reads → drop old. Or gh-ost / pt-online-schema-change for MySQL.
"How to build read replicas without consistency surprise?"Read-your-writes: route writes + immediate reads to leader; eventual reads to replicas. Or session-pinning.
