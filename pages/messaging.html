Messaging & Event-Driven Systems

The backbone of decoupled, scalable architectures. Get the semantics right and you eliminate entire classes of failure.

Mental model

  • Messaging trades immediacy for durability and decoupling. The async path adds latency and an extra failure surface (the broker), in exchange for buffering, fan-out, and replay.
  • Pick semantics first: at-most-once / at-least-once / exactly-once. Real "exactly-once" requires idempotent producers + transactional consumers (or outbox + idempotent handler).
  • Order is partition-local in Kafka. Partition by the entity that needs ordering (user_id, account_id) — never by random.
  • Outbox + idempotent handler is the modern at-least-once recipe — no 2PC, partition-tolerant, simple to operate.
  • Always have a DLQ + redrive tooling. Messages that fail repeatedly need human attention, not silent drops.

Why messaging?

  • Decoupling — producers and consumers evolve independently.
  • Buffering — absorbs traffic spikes; smooths downstream load.
  • Fan-out — one event → many consumers.
  • Async workflows — long-running tasks without blocking the request path.
  • Audit / replay — event log = source of truth.

Delivery semantics

  • At-most-once — fastest; messages can be lost. Useful for metrics/logs you can sample.
  • At-least-once — default in most systems; consumers must be idempotent.
  • Exactly-once — needs both idempotent producer and transactional consumer (Kafka transactions, or Outbox + idempotent handler).

Kafka mental model

  • Topic = log of messages, divided into partitions.
  • Order is guaranteed within a partition, never across partitions.
  • Partition key chooses partition (hash). Pick a key that preserves required ordering (e.g. user_id).
  • Consumer group — partitions split across consumers; one partition → one consumer at a time.
  • Offsets committed by consumer; replay = reset offset.
  • Retention by time or size; Kafka can be a storage layer for days/weeks.

Patterns

Transactional outbox

// In the same DB transaction that writes business state, also INSERT into outbox table.
// A relay process polls outbox → publishes to broker → marks rows as sent.
// This gives at-least-once event delivery without 2PC.
BEGIN;
INSERT INTO orders ...;
INSERT INTO outbox(event_id, payload, status) VALUES (..., 'pending');
COMMIT;

Saga (long-running transaction)

  • Choreography — each service reacts to events and emits new ones. Decentralized, hard to debug.
  • Orchestration — central coordinator (Temporal, Cadence) drives steps and compensations. Easier to reason about.
  • Each step has a forward action and a compensating action. State is durable.

CQRS & event sourcing

  • CQRS: separate write model (commands) and read models (queries). Read models built from events.
  • Event sourcing: state = fold(events). Replay to rebuild; snapshots for performance.
  • Powerful but expensive — adopt where audit/replay/temporal queries justify it.

Dead-letter queues (DLQ)

  • After N failed retries, ship the message to a DLQ. Alert and inspect — don't silently drop.
  • Provide tooling to redrive from DLQ after fix.

Backpressure

  • Bounded queues; pause/slow producers when consumers fall behind.
  • Kafka: consumer lag metric is your primary backpressure signal.
  • RabbitMQ: prefetch count limits unacked messages per consumer.

Idempotency in Go

// Use a deduplication store keyed by message ID with a TTL.
type Handler struct{ store IdemStore }

func (h *Handler) Handle(ctx context.Context, m Message) error {
    ok, err := h.store.PutIfAbsent(ctx, m.ID, time.Hour)
    if err != nil { return err }
    if !ok    { return nil } // already processed
    return h.process(ctx, m)
}

Schema management

  • Use a schema registry (Confluent, Apicurio) with Avro/Protobuf.
  • Compatibility modes: backward, forward, full. Pick based on rollout order of producers/consumers.
  • Never reuse a field tag in Protobuf. Add new fields, deprecate old.

Observability for messaging

  • Track lag, throughput, error rate, retry count, DLQ depth.
  • Propagate trace IDs across producer → broker → consumer for end-to-end traces.
  • Measure end-to-end event latency (produce → consume) at p95/p99.

Anti-patterns — what NOT to say

  • "RPC then publish" — if the publish fails after the RPC, you've leaked state. Use outbox.
  • "We'll have exactly-once." Without idempotent producer + transactional consumer, you have at-least-once.
  • Random partition keys — kills ordering guarantees.
  • Drinking from the firehose without backpressure — consumer crashes under load.
  • Silent message drops (no DLQ) — bugs hide indefinitely.
  • Reusing Protobuf field tags — instantly breaks compat across deployed versions.
  • Treating Kafka as a database when retention is too short.

Senior interviewer follow-ups

ProbeWhat they want to hear
"How do you guarantee at-least-once between two services?"Transactional outbox + idempotent consumer with dedup table.
"Why not 2PC across DB and broker?"Coordinator failure halts the whole system; doesn't survive partitions; high latency.
"How does Kafka give exactly-once?"Producer with transactional ID + idempotent send; consumer commits offsets and downstream effects in one transaction.
"What's the cost of partition rebalancing?"Brief consumer downtime; in-progress messages re-delivered on rebalance.
"Choreography vs orchestration for sagas?"Choreography = decentralized events; orchestration = central state machine (Temporal, Cadence). Orchestration is easier to debug.
"How to fan out a single event to 50 consumers?"One topic with 50 consumer groups (Kafka), or pub/sub topic with subscriptions.
"Schema compat strategies?"Backward (consumers upgraded after producers) / forward / full. Pick based on rollout order.
