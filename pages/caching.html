Caching

The single highest-leverage performance lever — and the single most common source of bugs in distributed systems.

Mental model

  • Cache where the read pattern is skewed (a few keys account for most traffic). Uniform random reads don't benefit much.
  • The trade-off triangle: latency ↓, throughput ↑, consistency ↓. Pick which to sacrifice based on the data.
  • Six failure modes to enumerate: thundering herd, stampede on restart, hot key, big key, cache penetration, cache avalanche.
  • Invalidation is the unsolved problem. TTL is your default; explicit invalidation is best-effort; versioned keys sidestep races.
  • Multi-tier when justified: process-local LRU (sub-µs) → shared Redis (~1 ms) → DB (10+ ms).

Where to cache

  1. Client — HTTP cache headers (Cache-Control, ETag, max-age).
  2. Edge / CDN — static assets and cacheable APIs.
  3. API gateway — short TTL for hot endpoints.
  4. In-process — per-instance LRU/LFU for very hot keys (sub-µs).
  5. Distributed — Redis/Memcached cluster shared across replicas.
  6. Materialized view in DB — pre-computed expensive joins.

Caching patterns

PatternHowTrade-off
Cache-aside (lazy)App reads cache → on miss reads DB → writes cache. Writes go to DB and invalidate cache.Simple, but cold start storms and possible stale data.
Read-throughApp reads cache; cache transparently fetches from DB on miss.Hides loader logic; requires cache to know schema.
Write-throughEvery write goes to cache + DB synchronously.Consistent; doubles write latency.
Write-back (write-behind)Write to cache; async flush to DB.Fastest writes; risk of data loss on cache crash.
Refresh-aheadBackground refresh before TTL expires.Smooths latency; wastes work on cold keys.

Eviction policies

  • LRU — drop the least-recently-used. Good for recency-skewed workloads.
  • LFU — drop the least-frequently-used. Good for stable popularity.
  • ARC — adaptive replacement; balances recency vs frequency.
  • TTL-based — keys expire after fixed time; pair with LRU.
  • 2Q / LIRS / TinyLFU — modern algorithms with high hit ratios.

Consistency

  • Cache invalidation is famously hard. Strategies:
    • TTL only (simplest, may serve stale).
    • Explicit invalidate on write (race: stale read may write back).
    • Versioned keys (user:42:v17) — new writes change the version.
    • Write-through + read-through with single source.
  • Negative caching: cache "not found" with a short TTL to avoid hammering DB.
  • Soft TTL / hard TTL: serve stale during soft, force refresh after hard.

Failure modes & how to defend

  • Thundering herd on miss → use singleflight, request coalescing, jittered TTLs.
  • Cache stampede after restart → warm cache before opening LB traffic; use refresh-ahead.
  • Hot key → split key across many physical keys, or replicate in local LRU.
  • Big key (huge value) → split or compress; avoid blocking Redis with O(N) ops.
  • Cache penetration (queries that always miss) → negative cache + bloom filter at the front.
  • Cache avalanche (mass TTL expiry at once) → jitter the TTLs; staggered warm-up.

Go: singleflight in one snippet

import "golang.org/x/sync/singleflight"

var g singleflight.Group

func getUser(ctx context.Context, id string) (*User, error) {
    v, err, _ := g.Do("user:"+id, func() (any, error) {
        if u, ok := cache.Get(id); ok { return u, nil }
        u, err := db.Get(ctx, id)
        if err != nil { return nil, err }
        cache.Set(id, u, 5*time.Minute)
        return u, nil
    })
    if err != nil { return nil, err }
    return v.(*User), nil
}

Redis specifics worth knowing

  • Data types: strings, hashes, lists, sets, sorted sets, streams, HyperLogLog, geo, bitmaps.
  • Sorted sets enable leaderboards, rate limiters, time-based queues.
  • Lua scripts are atomic — perfect for compare-and-set semantics (token-bucket limiter).
  • Persistence: RDB (snapshot) + AOF (append-only). Tune for RPO needs.
  • Cluster mode shards by hash slot (16384 slots). Use hash tags {user:42}:cart to keep related keys co-located.
  • SCAN instead of KEYS in production (non-blocking).

Cache sizing

  • Working set ≈ hot key count × avg value size × replication.
  • Target hit ratio: 80%+ for read-heavy systems; measure with cache hit/miss metrics.
  • Memory headroom of 25% for fragmentation and Redis overhead.

Anti-patterns — what NOT to say

  • "Just cache it." Always specify: which keys, what TTL, how invalidated, what's the failure plan.
  • Same TTL for every key — leads to mass expiry (avalanche). Add jitter.
  • Caching writes-through without considering DB-cache divergence on write failure.
  • Caching mutable user-personalized data with shared keys — leak across users.
  • Read-modify-write back to cache without versioning — lost updates.
  • Reaching for Redis when an in-process LRU is the right answer (single-instance, micro-cache).

Senior interviewer follow-ups

ProbeWhat they want to hear
"How do you avoid stampede on a popular cold key?"Singleflight on the loader; one in-flight request fans out; or pre-warm.
"How do you invalidate a cache safely?"Versioned keys (writes change version), or write-through, or TTL + tolerable staleness.
"What's cache penetration vs avalanche vs stampede?"Penetration = misses-that-always-miss (DoS); avalanche = mass simultaneous expiry; stampede = many concurrent loads on one key.
"Redis cluster — how do you keep related keys together?"Hash tags {user:42}:cart route by tagged substring.
"How do you size for hit ratio?"Sample key access frequency; the long tail's working set sets your memory floor for a target hit ratio.
"What's negative caching?"Cache "not found" with a short TTL to throttle DB on missing-key DoS.
"Cache vs CDN — when each?"CDN = geographic edge for static / cacheable HTTP; app-cache = within DC for dynamic data.

Self-check

  • I can explain cache-aside vs read-through vs write-through vs write-back.
  • I can use singleflight to prevent stampede.
  • I know LRU/LFU/TTL trade-offs.
  • I can size a cache from working-set + hit-ratio target.
  • I can use versioned keys to invalidate safely.
  • I know Redis hash tags for cluster co-location.
