Sorting

Knowing when not to sort is half the battle. Master quickselect, counting/radix sort, custom comparators, and stable vs unstable trade-offs.

Mental model

  • Sorting is rarely the goal — it's a setup. Ask: do I really need a total order, or just the top-K, the median, partitioning, or counting?
  • Time floor: comparison-based sorting is Ω(n log n). Counting / radix beat it when the value range is small or the data is digit-limited.
  • Stable when ties matter; unstable otherwise (it's faster).
  • For top-K: heap of K (streaming), or quickselect (offline). Don't sort the whole array.
  • Custom comparators: in Go, return int from slices.SortFunc (negative / zero / positive) or bool from sort.Slice's less.

Go standard library

sort.Ints(a)
sort.Strings(ss)
sort.Slice(items, func(i, j int) bool { return items[i].x < items[j].x })
sort.SliceStable(items, less)        // preserves relative order of equals
sort.Sort(sort.Reverse(sort.IntSlice(a)))

// generics (1.21+)
slices.Sort(a)
slices.SortFunc(items, func(a, b Item) int { return cmp.Compare(a.x, b.x) })
  • sort.Slice/slices.Sort use pattern-defeating quicksort (pdqsort) — O(n log n) worst case in modern Go.
  • Use SortStable only when ties matter; it's slightly slower.

Counting sort (small integer range)

// O(n + k). Use when values are bounded (e.g. 0..1000).
func countingSort(a []int, k int) {
    c := make([]int, k+1)
    for _, v := range a { c[v]++ }
    i := 0
    for v, cnt := range c {
        for ; cnt > 0; cnt-- { a[i] = v; i++ }
    }
}

Quickselect — k-th smallest in average O(n)

func quickselect(a []int, k int) int {  // k is 0-indexed
    lo, hi := 0, len(a)-1
    for lo < hi {
        p := partition(a, lo, hi)
        if p == k { return a[p] }
        if p < k { lo = p + 1 } else { hi = p - 1 }
    }
    return a[lo]
}
func partition(a []int, lo, hi int) int {
    pivot := a[hi]
    i := lo
    for j := lo; j < hi; j++ {
        if a[j] < pivot { a[i], a[j] = a[j], a[i]; i++ }
    }
    a[i], a[hi] = a[hi], a[i]
    return i
}
Quickselect's worst case is O(n²). Randomize the pivot or use median-of-three to avoid attacker-chosen inputs.

Merge sort (used in linked-list sort, count inversions)

func mergeSort(a []int) []int {
    if len(a) < 2 { return a }
    m := len(a) / 2
    l := mergeSort(append([]int(nil), a[:m]...))
    r := mergeSort(append([]int(nil), a[m:]...))
    return merge(l, r)
}

Anti-patterns — what NOT to say

  • "Sort and pick K-th." That's O(n log n) when O(n) average is possible (quickselect, bucket).
  • Comparator that doesn't define a total order — produces undefined behavior, not just wrong order.
  • Stable sort by reflex when ties don't matter (slower).
  • In-place sorting a slice you've sub-sliced — original gets reordered too.
  • Sorting strings with byte comparison when you need locale / Unicode order — use golang.org/x/text/collate.

Senior interviewer follow-ups

ProbeWhat they want to hear
"How would you sort 100GB of integers on a 16GB machine?"External merge sort: chunk into RAM-sized runs, sort, write to disk, k-way merge with a heap.
"Why O(n log n) lower bound for comparison sorts?"Decision tree must distinguish n! permutations; tree depth ≥ log₂(n!) ≈ n log n.
"When is radix sort better than quicksort?"Small fixed-width integers / strings; very large n. Constant factor matters.
"Quickselect's worst case?"O(n²). Use randomized or median-of-medians pivot for guaranteed O(n).
"How do you sort an immutable input?"Sort a slice of indices via custom less-fn that reads the input.
"Find K closest points to origin in O(n) avg?"Quickselect on squared distance; or bucket sort if range is known.
"Stable vs unstable — which does Go use?"Pdqsort (unstable) for sort.Slice; sort.SliceStable for stable.
