System Design Case Studies

Six end-to-end walkthroughs at the senior bar — capacity math, schemas, sequence diagrams (ASCII), trade-offs, and failure modes. Read one a day.

Mental model

  • Six classic flavors: URL shortener (write throughput + KV), rate limiter (atomicity in Redis), news feed (fan-out trade-off), real-time chat (long-lived connections + ordering), distributed cache (sharding + replication), search/typeahead (offline build + serve from RAM).
  • Pacing for 45 min: 5 reqs · 5 capacity · 5 high-level · 10 data + APIs · 15 deep dives · 5 trade-offs/scale.
  • Always estimate first; the math drives every later decision.
  • Always pick a sharding key with reasoning; if you don't, the interviewer will probe.

How to read these

  1. Cover the answer; spend 5 minutes drafting your own approach against the prompt.
  2. Compare with the walkthrough — note where your design diverged.
  3. Practice the verbal: 7-step framework, capacity math, then deep dive.

1 · URL Shortener (TinyURL / bit.ly)

Requirements

  • POST /shorten {url} → short code. GET /{code} → 301 to long URL.
  • ~100M new URLs/day, 10:1 read:write, 5-year retention, p99 redirect < 50 ms global.
  • Codes must be short (~7 chars), unguessable enough, never collide.

Capacity

writes_per_day := 100_000_000
reads_per_day  := writes_per_day * 10        // 10^9
avg_write_qps  := writes_per_day / 86_400    // ~1.2 k
avg_read_qps   := reads_per_day  / 86_400    // ~12 k
peak_read_qps  := avg_read_qps * 3           // ~36 k

bytes_per_row  := 500                        // url + meta
storage_5y     := writes_per_day * 365 * 5 * bytes_per_row // ~91 TB raw

API

  • POST /v1/urls body {url, custom?, ttl?} → {code, short_url}. Idempotency-Key header.
  • GET /{code} → 301 redirect (cacheable). 404 if expired/missing.
  • GET /v1/urls/{code}/stats → click counts.

Code generation strategies

ApproachProsCons
Auto-increment ID → base62No collisions, sequentialGuessable; needs a single counter or sharded counters
Snowflake ID → base62Distributed, time-orderedLength grows; mildly guessable
Random 7-char + collision retryUnguessable, simpleCollision check needs a fast read (Bloom filter helps)
Pre-generate code poolConstant-time writeBackground filler; ops complexity

Architecture (ASCII)

           +-------+    +--------+    +-----------+    +-----------+
client --> |  CDN  |-->|   LB   |-->|  shortenr  |-->| Postgres  |
           +-------+    +--------+    +-----+-----+    +-----------+
                                            |  ^
                                            v  |
                                       +----+--+----+
                                       |   Redis    |  (hot code -> url)
                                       +------------+
                                            |
                                            v
                                       Kafka click stream -> analytics

Data model

CREATE TABLE urls (
  code        VARCHAR(10) PRIMARY KEY,
  long_url    TEXT        NOT NULL,
  user_id     BIGINT,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at  TIMESTAMPTZ
);
CREATE INDEX idx_urls_user_created ON urls(user_id, created_at DESC);

Read path

  1. CDN: cache 301 by code for short TTL (e.g. 60 s) — absorbs hot codes globally.
  2. App: GET code → Redis (LRU, ~95% hit) → Postgres on miss → fill cache.
  3. Async: emit click event to Kafka (don't block redirect).

Trade-offs & gotchas

  • Hot key (viral link) — CDN + Redis absorb it; rate-limit obvious abuse.
  • Phishing — moderation pipeline against URL safety APIs.
  • Custom slugs — unique constraint; reserve namespace for vanity codes.
  • Analytics consistency — at-least-once events; dedupe in the warehouse.

2 · Distributed Rate Limiter

Requirements

  • Per-API-key limits (e.g. 1000 req/min) across N stateless service nodes.
  • p99 limit-check < 5 ms; correctness must survive single-node loss.

Algorithm choice

  • Token bucket in Redis — allows bursts up to capacity; smooth refill.
  • Implement as a Lua script for atomicity (read state, refill, decide, write).
-- KEYS[1] = bucket key; ARGV = capacity, refill_rate, now_ms, requested
local b = redis.call('HMGET', KEYS[1], 'tokens', 'ts')
local tokens = tonumber(b[1]) or tonumber(ARGV[1])
local ts     = tonumber(b[2]) or tonumber(ARGV[3])
local delta  = (tonumber(ARGV[3]) - ts) * tonumber(ARGV[2]) / 1000
tokens = math.min(tonumber(ARGV[1]), tokens + delta)
local req = tonumber(ARGV[4])
local ok  = tokens >= req
if ok then tokens = tokens - req end
redis.call('HMSET', KEYS[1], 'tokens', tokens, 'ts', ARGV[3])
redis.call('PEXPIRE', KEYS[1], 60000)
return ok and 1 or 0

Architecture

  +-----------+   pre-filter   +-----------+
  |  service  |--------------->|   Redis   |  (1 round-trip per request)
  +-----------+                +-----------+
        |
        v
   downstream

Trade-offs

  • Hard limit requires a single Redis primary per key — use consistent hashing across N shards.
  • Approx limit via local token bucket + periodic sync (1–2% over-allow, much faster).
  • Failure mode — Redis down: fail-open (allow + alert) or fail-closed (deny) by policy.

3 · News Feed (Twitter / Instagram timeline)

Requirements

  • 200M DAU, p99 home-feed < 200 ms, eventual consistency OK.
  • Average user follows ~200, top 0.1% follow > 100k (celebrities).

The fan-out trade-off

  • Fan-out on write (push): when A tweets, copy tweet ID to each follower's timeline. Fast reads, expensive writes for celebrities.
  • Fan-out on read (pull): on read, query latest tweets from each followee, merge. Expensive reads, simple writes.
  • Hybrid (winner): push for normal users, pull for celebrities. Merge at read time.

Architecture

post tweet --> tweet svc --> Kafka(tweets) --> fanout workers --> Redis(timelines)
                                                  |
                                                  +--> (skip if celeb)
read feed --> feed svc --> Redis(my timeline) ++ pull(celeb tweets) --> merge --> rank --> cache

Storage

  • Tweets: Cassandra/Scylla, partition by tweet_id (Snowflake), wide rows for retweets/replies.
  • Social graph: dedicated graph store or Postgres for < 1B edges.
  • Timelines: Redis sorted set per user, capped at ~800 entries.
  • Media: object store (S3) + CDN; tweets store URLs only.

Senior follow-ups

  • Ranking — separate ranker service; features served from a feature store.
  • Cold cache — pull from tweets store and warm Redis on first read.
  • Backfill — when a user follows someone, async job inserts recent tweets into their timeline.
  • Deletes — soft-delete in tweet store; timelines self-heal as old IDs roll off.

4 · Real-time Chat (WhatsApp / Slack)

Requirements

  • 1B users, 50B msgs/day, 1:1 + group up to 1000, delivery < 200 ms p99 same-region.
  • At-least-once delivery; ordering per conversation.

Architecture

  client --(WebSocket/MQTT)--> gateway (sticky) --> chat svc --> Kafka(conv_id)
                                       ^                          |
                                       |                          v
                                       +-- presence/push       Cassandra(messages)

Key choices

  • Long-lived connection per device → gateway holds (user_id, conn). Use consistent hashing for routing.
  • Partition key = conversation_id in Kafka → preserves order per chat.
  • Storage: Cassandra row per message, partition (conv_id, bucket=day), clustering by msg_id desc.
  • Offline delivery: per-user inbox in KV with last_seen offset; client pulls deltas on reconnect.
  • Push notifications: APNs/FCM via separate worker reading from Kafka.

Hard problems

  • Group fan-out at 1000 members — write once to log, fan out to gateways via pub/sub; don't write 1000 inbox rows.
  • Read receipts & typing — ephemeral, broadcast through pub/sub, don't persist.
  • Encryption — E2E with Signal protocol; server stores ciphertext only.
  • Multi-region — pick a home region per conversation; replicate async; conflict resolution by msg_id.

5 · Distributed Cache (Memcached/Redis-cluster)

Requirements

  • 1M ops/s, p99 < 1 ms in same DC, 10 TB working set, eventual consistency on writes.

Design

  • Sharding by consistent hashing with virtual nodes (200/node) for balance.
  • Replication primary-secondary per shard; reads from primary by default, secondary for read-heavy keys.
  • Eviction LRU per shard; allkeys-lru for cache use; volatile-lru if mixed TTL.
  • Client picks shard via hash ring; reconnect on slot move.

Failure modes

  • Hot key — replicate the key across N shards; client jitters reads.
  • Stampede on miss — singleflight + jittered TTL + early-refresh at 80% TTL.
  • Node loss — failover to secondary; rebalance moves only K/N keys (consistent hashing).
  • Split brain — fence with a coordinator (etcd) holding shard ownership.

Capacity

working_set     := 10 * (1 << 40)            // 10 TB
mem_per_node    := 64 * (1 << 30)            // 64 GB
shards          := working_set / mem_per_node // ~160
nodes           := shards * 2                 // x2 for replication
ops_per_node    := 1_000_000 / shards         // ~6k ops/s — well within Redis

6 · Search Autocomplete (typeahead)

Requirements

  • Suggest top-10 completions for a prefix in < 50 ms p99.
  • Update top-K daily from query logs; freshness OK at 1-day lag.

Design

  • Build phase (offline): aggregate query logs by day → top-K per prefix → ship as a trie or Marisa-trie blob.
  • Serve phase: in-memory trie per node (RAM cheap; tries compress well). Lookup = O(prefix length).
  • Cache hot prefixes at the edge (CDN/Redis) with 5-minute TTL.
  • Personalization: blend global top-K with per-user top-K at request time.

Architecture

click logs --> spark/flink --> top-K aggregator --> S3 trie blob
                                                    |
serve nodes load blob on rollout                    v
client --> LB --> suggest svc(in-mem trie) --> (cache miss?) --> rebuild from blob

Trade-offs

  • Real-time freshness — needs a streaming counter (Count-Min Sketch) → expensive but necessary for trends.
  • Misspellings — BK-tree or Levenshtein automaton for fuzzy fallback.
  • Multilingual — shard tries per locale; serve from nearest region.

Other classics worth practicing

  • Web crawler — frontier (priority queue), URL dedupe (Bloom filter), politeness, robots.txt.
  • Notification system — channel routing, retries, DLQ, user preferences.
  • Payment system — idempotency, double-entry ledger, eventual consistency between services, reconciliation.
  • Ride-sharing — geo-index (S2/H3), driver-rider match, surge pricing.
  • Video streaming — chunked HLS/DASH, CDN, ABR ladder, encoding pipeline.
  • Metrics system — Prometheus-style: scrape, time-series store, downsampling, alerting.

The verbal pacing (45-min slot)

  1. 0–5 min — clarify requirements; write functional + non-functional on the board.
  2. 5–10 min — capacity math; QPS, storage, bandwidth.
  3. 10–15 min — high-level architecture; one box per service.
  4. 15–25 min — data model + APIs; sharding key called out.
  5. 25–40 min — pick 2 components and deep-dive (the part the interviewer leans into).
  6. 40–45 min — failure modes, scaling to 10× / 100×, what you'd do differently with more time.
