System Design Building Blocks

The 12 components every senior-level design draws from. Know what each does, its failure modes, and how to scale it.

Mental model

  • The 12 building blocks: DNS · CDN · Load Balancer · API Gateway · Application Servers · Cache · Databases · Queues · Search · Object Storage · Stream Processing · Batch & Schedulers.
  • Every senior design picks the right block, sizes it, lists its failure modes, and articulates how it scales.
  • Layer locality matters: closer-to-user (CDN, edge cache) handles read amplification; closer-to-data (DB, search) handles write amplification.
  • Every block needs an SLO; each block's SLO chains into the next.

1. DNS

  • Translates hostnames to IPs. TTL controls cache; use low TTL for failover, high TTL for performance.
  • GeoDNS / latency-based routing (Route 53, NS1) steers users to nearest region.

2. CDN

  • Edge cache for static + cacheable dynamic content; reduces RTT and origin load.
  • Cache-control headers, ETags, signed URLs for private content.
  • Edge compute (Cloudflare Workers, Lambda@Edge) for personalization at the edge.

  • Health checks + outlier detection + connection draining.
  • Sticky sessions vs stateless: prefer stateless with external session store.

4. API Gateway

  • Single entry point: authn, rate limiting, request transformation, routing to microservices.
  • Pairs with a service mesh (Envoy/Istio/Linkerd) for east-west traffic.

5. Application servers (in Go)

  • Stateless replicas behind LB; horizontal scale by adding pods.
  • One graceful-shutdown hook; reuse http.Client; bound goroutines.
  • Use a worker pool / errgroup for fan-out to downstream services.

6. Cache

  • Local (per-instance LRU) → distributed (Redis/Memcached) → CDN.
  • Eviction: LRU, LFU, TTL, ARC. Pick based on workload (recency vs frequency).
  • Write patterns: read-through, write-through, write-back, cache-aside (most common).
  • Invalidation strategies: TTL only, explicit invalidate, key versioning, write-through with tags.
  • Stampede protection: singleflight, request coalescing, jittered TTLs, probabilistic early refresh.

7. Databases

See the deep dive. Headlines:

  • OLTP (PG, MySQL) for transactions; OLAP (Snowflake, BigQuery) for analytics.
  • Wide-column (Cassandra, ScyllaDB) for high write throughput.
  • Document (Mongo, DynamoDB) for flexible schemas; KV (Redis) for hot paths.
  • Graph (Neo4j) when traversals dominate.

8. Queues & event brokers

  • Kafka — high-throughput partitioned log; ordering within a partition.
  • RabbitMQ — flexible routing, lower throughput, easier semantics.
  • SQS / Pub/Sub — managed, simple, fan-out via topics.
  • Delivery: at-most-once, at-least-once, exactly-once (idempotency + transactional outbox).

9. Search

  • Elasticsearch / OpenSearch — inverted index + scoring (BM25).
  • Index pipeline: source DB → CDC → indexer → ES; control reindex windows.
  • Faceted search, autocomplete (edge n-grams), typo tolerance (fuzzy queries).

10. Object & blob storage

  • S3 / GCS / Azure Blob — 11 nines durability, eventual consistency on listing (S3 now strong).
  • Pre-signed URLs to upload/download directly from clients.
  • Lifecycle rules to tier to colder storage automatically.

11. Stream processing

  • Kafka Streams / Flink / Spark Streaming for windowed aggregations, joins, CEP.
  • Patterns: tumbling, sliding, session windows. Watermarks for late data.

12. Batch & schedulers

  • Airflow / Argo / Temporal for workflows and retries.
  • Idempotent jobs + checkpoints + DLQs.

Cross-cutting: load balancing strategies

  • Round-robin · weighted · least-connections · least-time
  • Consistent hashing (sticky to a backend without sessions; minimal remap on add/remove)
  • Power-of-two-choices: sample 2 backends, pick the less-loaded — near-optimal with O(1) overhead

Replication & partitioning

  • Replication: leader-follower (PG, MySQL), multi-leader (Cosmos), leaderless (Dynamo/Cassandra with quorum R+W>N).
  • Partitioning (sharding): by hash (uniform but range queries hard), by range (range queries easy but hot spots), by directory (lookup service).
  • Rebalancing: fixed virtual buckets per node (Cassandra vnodes, Redis hash slots) to avoid massive reshuffles.

Networking essentials

  • TCP vs UDP; HTTP/1.1 vs HTTP/2 (multiplexing) vs HTTP/3 (QUIC over UDP).
  • gRPC for service-to-service (Protobuf, streaming).
  • WebSockets / SSE for server push.
  • Connection pooling (always), keep-alive (almost always).

Anti-patterns — what NOT to say

  • "Add a load balancer" without specifying L4 vs L7.
  • "Cache it" without picking local-LRU vs Redis vs CDN.
  • "Sticky sessions" — always justify; usually a smell.
  • Skipping the API gateway and exposing services directly to clients.
  • Treating object storage as a database (no transactions, no queries).
  • Ignoring DNS as a part of failover plans.

Self-check

  • I can name all 12 building blocks and what each does.
  • I can pick L4 vs L7 load balancer with reasoning.
  • I can explain consistent hashing and virtual nodes.
  • I know power-of-two-choices load balancing.
  • I can pick the right cache layer (local / distributed / CDN) per workload.
  • I can pick the right DB family (relational / document / wide-column / KV / graph / search).
  • I know HTTP/1.1 vs 2 vs 3 differences.
  • I can use signed URLs for direct uploads to object storage.
  • I can describe stream processing windows (tumbling, sliding, session).
  • I can describe each block's failure modes and recovery.
