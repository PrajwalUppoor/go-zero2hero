Tries (Prefix Trees)

Anything with prefixes, autocomplete, word dictionaries or XOR-max-pair lives here. O(L) per op where L is the word length.

Mental model

  • Trigger words: prefix, autocomplete, dictionary of words, max XOR pair, longest common prefix, word break with many words.
  • Each node represents a prefix; children correspond to next characters. isWord marks a complete word.
  • Time complexity: O(L) per insert / search, O(N·L) total to build, where L is max word length.
  • Space is the killer: O(Σ N·L) worst case. Use a map of children for big alphabets, an array for fixed alphabets.
  • Two specialized variants: binary trie (bits, max-XOR pair); compressed trie / radix tree (path-compressed; used in routers, IP lookups, file systems).

Reference implementation

type Trie struct {
    children [26]*Trie
    isWord   bool
}

func (t *Trie) Insert(w string) {
    cur := t
    for i := 0; i < len(w); i++ {
        c := w[i] - 'a'
        if cur.children[c] == nil { cur.children[c] = &Trie{} }
        cur = cur.children[c]
    }
    cur.isWord = true
}

func (t *Trie) find(w string) *Trie {
    cur := t
    for i := 0; i < len(w); i++ {
        c := w[i] - 'a'
        if cur.children[c] == nil { return nil }
        cur = cur.children[c]
    }
    return cur
}

func (t *Trie) Search(w string)      bool { n := t.find(w); return n != nil && n.isWord }
func (t *Trie) StartsWith(p string)  bool { return t.find(p) != nil }
  • [26]*Trie children for lowercase. Use a map for big/Unicode alphabets.
  • Add metadata (count, list of indexes) per node for autocomplete or top-K suggestions.

Word Dictionary with wildcard '.'

type WordDict struct{ root *Trie }

func (d *WordDict) Search(w string) bool {
    var dfs func(n *Trie, i int) bool
    dfs = func(n *Trie, i int) bool {
        if n == nil { return false }
        if i == len(w) { return n.isWord }
        c := w[i]
        if c == '.' {
            for _, ch := range n.children { if ch != nil && dfs(ch, i+1) { return true } }
            return false
        }
        return dfs(n.children[c-'a'], i+1)
    }
    return dfs(d.root, 0)
}

Autocomplete with frequency

  • Store top-K (word, freq) pairs at every node that lies on any of its descendants' top-K paths.
  • On query: descend to prefix node, return its top-K cache. O(L + K) per query.
  • Update: on insertion, walk up from leaf updating top-K caches.

Anti-patterns — what NOT to say

  • "I'll use a hashmap of all prefixes." Works but balloons to O(N·L²) space — trie shares prefixes.
  • Recursive autocomplete that re-walks the entire subtree per query — cache top-K at each node instead.
  • Overusing map[byte]*Trie for ASCII — [26]*Trie is faster and tighter.
  • Forgetting to delete nodes when removing words — memory grows monotonically. Mark isWord = false and prune empty nodes upward.
  • Using a trie when a hashmap of full words is simpler and the workload is exact-match lookup.

Senior interviewer follow-ups

ProbeWhat they want to hear
"Trie vs hashmap for word lookup?"Hashmap = O(L) hashing then O(L) compare; trie = O(L) node walks but supports prefix queries naturally.
"How does Word Search II benefit from a trie?"Many candidates share prefixes; we descend the trie in lock-step with the board DFS, pruning whole branches.
"Max XOR of two numbers — why a binary trie?"For each x, walk the trie picking the opposite bit when possible; greedy gives the maximum XOR.
"Memory cost of a trie?"Up to O(N·L·Σ) where Σ is alphabet size; mitigations: child maps, radix/Patricia compression, suffix automaton.
"Aho–Corasick — when?"Multi-pattern matching in text. Build trie + failure links; scan text once in O(text + matches).
