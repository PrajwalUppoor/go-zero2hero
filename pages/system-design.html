System Design Fundamentals

The vocabulary, mental models, and math you need to reason about any distributed system in 45 minutes.

Mental model

  • Drive the interview: clarify, estimate, sketch, deepen, trade off, summarize. Senior candidates lead the conversation rather than wait for prompts.
  • Three numbers to anchor everything: 100 ns (memory) · 500 µs (DC RTT) · 150 ms (cross-continent RTT). Order of magnitude differences drive every architectural choice.
  • Trade-offs, not religions: SQL vs NoSQL, cache-aside vs write-through, push vs pull, monolith vs microservices — every choice is contextual.
  • Operational thinking is what separates seniors: failure modes, observability, rollouts, capacity, on-call. Don't end with a happy-path diagram.
  • SLO-driven design: pick targets first (e.g. p99 ≤ 100 ms, 99.9% availability) then work backwards through the stack.

The 7-step interview framework

  1. Requirements — functional (5–6) + non-functional (latency, availability, durability, consistency, scale).
  2. Capacity estimation — QPS, storage, bandwidth, working set.
  3. API design — REST/gRPC endpoints; idempotency keys; pagination.
  4. High-level architecture — boxes & arrows: client → LB → service → cache → DB → async pipeline.
  5. Data model & storage — schemas, indexes, choice of database, sharding key.
  6. Deep dives — pick 2–3 components and stress-test (hotspots, failure modes, observability).
  7. Trade-offs & follow-ups — what would you do at 100x scale? What about cross-region?

Capacity-estimation recipes

// Daily QPS → peak
dau := 100_000_000
reqs_per_user_day := 20
total_per_day := dau * reqs_per_user_day                  // 2 * 10^9
avg_qps := total_per_day / 86_400                          // ~ 23,000
peak_qps := avg_qps * 3                                    // ~ 70,000

// Storage (5-year retention, 2 KB per event)
events_per_day := total_per_day                            // 2 * 10^9
bytes_per_day  := events_per_day * 2_000                   // 4 TB / day
total_storage  := bytes_per_day * 365 * 5                  // ~ 7.3 PB raw
// Add replication factor 3 → ~22 PB
  • 1 KB ≈ 10³ B · 1 MB ≈ 10⁶ B · 1 GB ≈ 10⁹ B.
  • 1 day = 86,400 s ≈ 10⁵ s.
  • Peak = 2–10× average; design for 3× unless told otherwise.

Latency, throughput, availability, durability

  • Latency — time per request. Care about p50/p95/p99/p999 not just averages.
  • Throughput — requests/sec the system can sustain. Limited by the slowest component.
  • Availability — % uptime. “3 nines” = 8.76h/yr down, “4 nines” = 52m/yr, “5 nines” = 5m/yr.
  • Durability — probability data persists once acknowledged. 11 nines = S3-class.

CAP & PACELC

  • CAP: under a network partition, choose Consistency or Availability — you can't have both.
  • PACELC: even when there's no partition (Else), trade Latency vs Consistency.
  • Real systems are tunable per request (quorum reads/writes, read replicas, consistency levels).

Consistency models, from strong to weak

  1. Linearizable — appears as a single global timeline (etcd, single-leader writes).
  2. Sequential — all observers see the same order, not necessarily real-time.
  3. Causal — preserves cause-and-effect order (Dynamo with vector clocks).
  4. Read-your-writes — see your own writes immediately.
  5. Monotonic reads — never see time go backward.
  6. Eventual — replicas converge if writes stop.

Bottlenecks you'll be probed on

  • Hot keys — one key gets disproportionate traffic. Mitigate with replication, key sharding (k:bucket), local caching.
  • Thundering herd on cache miss — use singleflight, request coalescing, locking, jittered TTLs.
  • Cascading failures — circuit breakers, bulkheads, load shedding, timeouts everywhere.
  • Tail latency — hedged requests, redundant reads, isolating noisy neighbors.
  • Backpressure — bounded queues, drop-newest vs drop-oldest, slow-start.

Reliability patterns

  • Timeouts → retries (with exponential backoff + jitter) → circuit breaker → fallback.
  • Idempotency keys for safe retries.
  • Health checks (liveness vs readiness).
  • Graceful shutdown: drain LB, finish in-flight, close.
  • Chaos testing: kill instances in staging, verify SLOs.

Observability

  • Metrics (Prometheus): RED method (Rate, Errors, Duration) for services; USE (Utilization, Saturation, Errors) for resources.
  • Logs: structured (JSON), with trace IDs propagated end-to-end.
  • Traces: OpenTelemetry; sample ~1% in prod, 100% for errors.
  • SLI / SLO / SLA: SLIs measured, SLOs targeted, SLAs contractual. Error budgets drive release decisions.

Security checklist (mention even if not asked)

  • TLS everywhere; mTLS internally.
  • AuthN (OIDC) vs AuthZ (RBAC/ABAC).
  • Secrets in a vault, rotate periodically.
  • Input validation + parameterized queries (SQLi).
  • Rate limiting + bot protection at the edge.
  • Encryption at rest (envelope encryption) + in transit.
  • Audit logs for sensitive actions.

Anti-patterns — what NOT to say

  • "Just add a cache." Caches add another consistency surface; explain invalidation, TTL, miss storms.
  • "Microservices for scale." Microservices solve team scale, not request scale, at the cost of operational overhead.
  • "We'll be eventually consistent." Eventually consistent where? Spell out the affected reads.
  • "Sharding fixes hot keys." Hot keys live within a shard; you need replication or load-aware splitting.
  • Drawing a diagram with no failure modes — every box and arrow has a fail mode.
  • Picking technologies without justifying with the requirements (no "pick Cassandra because scale").

Senior interviewer follow-ups

ProbeWhat they want to hear
"Walk me through what happens at 10x scale."Identify next bottleneck (DB writes, cache memory, network), the mitigation (sharding, replication, queueing), and the new failure mode it introduces.
"How do you handle a noisy neighbor / hot key?"Replicate hot keys to multiple replicas; key-bucket suffix; client-side caching; rate-limiting per key.
"What's your read latency budget?"Pick a target (p99 ≤ X ms) and show the per-component allocation (LB 1 ms, app 10 ms, cache 0.5 ms, fallback DB 30 ms).
"What about cross-region?"Active-active with conflict resolution OR active-passive with manual failover. RPO and RTO trade-offs.
"How do you roll out safely?"Feature flags, canaries (1%, 10%, 50%, 100%), automatic rollback on SLO breach.
"How do you avoid a thundering herd?"Singleflight, jittered TTLs, request coalescing at cache layer, stale-while-revalidate.
"Idempotency design?"Client-supplied UUID per mutation; server stores result for N hours; same key → same response.
"How do you size a cache?"Working-set estimation × hit-rate target. Smaller hot tier with eviction beats one giant cache.

Self-check

  • I can drive a 45-min interview through the 7-step framework.
  • I know latency numbers (memory, SSD, DC RTT, cross-continent RTT) by heart.
  • I can do back-of-envelope QPS and storage estimates.
  • I can explain CAP and PACELC with a concrete example.
  • I can name 5 consistency models from strongest to weakest.
  • I can describe a thundering herd and 3 mitigations.
  • I can describe hot-key mitigation and load shedding.
  • I know how to design idempotent mutations.
  • I can explain SLI/SLO/SLA + error budgets.
  • I can describe RED and USE methods for monitoring.
  • I can outline a graceful-shutdown procedure.
  • I can describe canary rollouts and automatic rollback.
