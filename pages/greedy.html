Greedy

The hardest pattern to justify, the easiest to code. Lean on sorting + an exchange argument.

Mental model

  1. Sort by one key (start, end, ratio, value, deadline).
  2. Scan once, maintain a running state, make the locally-optimal choice.
  3. Prove with an exchange argument: swapping to optimal order does not decrease value.
  • Greedy works when the problem has the greedy-choice property (a local optimum leads to a global one) and optimal substructure.
  • Five greedy schools:
    • Sort by end-time — interval scheduling, balloons, non-overlapping.
    • Sort by start, sweep — meeting rooms, merge intervals.
    • Heap-based — task scheduler, reorganize string, scheduling with deadlines.
    • Two-pointer greedy — boats to save people, container with most water.
    • Custom comparator — largest number, queue reconstruction.
  • If you can't argue why greedy works in 30 seconds, switch to DP. Knapsack-with-arbitrary-weights is the classic counterexample.

Interval scheduling — maximum non-overlap

// Sort by end time; greedily pick the earliest-ending compatible interval.
sort.Slice(iv, func(i, j int) bool { return iv[i][1] < iv[j][1] })
count, end := 0, math.MinInt
for _, x := range iv {
    if x[0] >= end { count++; end = x[1] }
}

Jump Game

func canJump(a []int) bool {
    reach := 0
    for i, v := range a {
        if i > reach { return false }
        if i + v > reach { reach = i + v }
    }
    return true
}

// Jump Game II — minimum jumps to reach end.
func jump(a []int) int {
    jumps, end, far := 0, 0, 0
    for i := 0; i < len(a)-1; i++ {
        if i+a[i] > far { far = i + a[i] }
        if i == end { jumps++; end = far }
    }
    return jumps
}

Gas Station — start index

func canCompleteCircuit(gas, cost []int) int {
    total, tank, start := 0, 0, 0
    for i := range gas {
        d := gas[i] - cost[i]
        total += d; tank += d
        if tank < 0 { tank = 0; start = i + 1 }
    }
    if total < 0 { return -1 }
    return start
}

Largest Number (custom comparator)

func largestNumber(nums []int) string {
    s := make([]string, len(nums))
    for i, n := range nums { s[i] = strconv.Itoa(n) }
    sort.Slice(s, func(i, j int) bool { return s[i]+s[j] > s[j]+s[i] })
    if s[0] == "0" { return "0" }
    return strings.Join(s, "")
}

Task Scheduler (max-heap)

  • Greedy with a max-heap on remaining counts; refill cooled tasks after n ticks.

When greedy fails

If you can't argue an exchange / matroid / cut property, DP is safer. Counter-examples (e.g. unbounded knapsack with weird weights) burn naïve greedy.

Anti-patterns — what NOT to say

  • "Greedy probably works." Without an argument, you'll be wrong on adversarial inputs.
  • Sorting by the wrong key (start vs end) — interval scheduling diverges sharply on the choice.
  • Assuming greedy + DP are the same. Greedy commits; DP explores.
  • Mixing greedy choices across phases (e.g. sort one way, then change the criterion mid-loop).

Senior interviewer follow-ups

ProbeWhat they want to hear
"Why sort by end time, not start time, for max non-overlapping intervals?"Earliest end leaves the most room for future intervals — exchange argument: replacing any picked interval with the earliest-ending compatible one preserves count.
"How do you prove a greedy correct?"Exchange argument (swap to optimal preserves value), or matroid (greedy on matroids gives optimum), or cut property (MST).
"Greedy vs DP for activity selection?"If only counting non-overlap → greedy. If maximizing weight → weighted interval scheduling = DP.
"Jump Game II — why is the level-by-level approach O(n)?"Each level represents one jump; the inner expansion only scans each index once.
"Gas Station — why reset start instead of trying every index?"If sum from i fails at j, no index in [i, j] can succeed (their suffixes are partial sums of i's failed run).
