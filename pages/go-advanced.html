Advanced Go

Memory model, escape analysis, GC, reflection, generics in depth, build flags, profiling — the senior-level material.

Mental model

  • The compiler is your ally. -gcflags=-m tells you what escaped to the heap; go test -bench + pprof tell you where the time really goes. Trust measurement, not intuition.
  • Garbage collection is concurrent and tri-color. STW pauses are sub-millisecond; tail latency comes from GC assist, not from pauses. Allocation reduction is the main GC tuning lever.
  • The Go memory model is "happens-before" via synchronization events — channels, mutexes, atomics, sync.Once, go fn(). Without one, no read is guaranteed to see another goroutine's write.
  • Reflection and unsafe exist for a reason but cost you type safety, speed, and clarity. Reach for them last, behind small, documented seams.

Memory model in 60 seconds

  • A read of a variable is not guaranteed to observe the latest write from another goroutine without a synchronization event.
  • Synchronization events: channel send/receive, sync.Mutex.Lock/Unlock, sync.Once.Do, atomic ops, and go fn() happens-before fn's first instruction.
  • The race detector (go test -race) finds violations in code paths actually exercised — coverage matters.

Escape analysis & inlining

// Compile with: go build -gcflags="-m -m" main.go
func sum(xs []int) int {           // xs may stay on stack
    s := 0
    for _, x := range xs { s += x }
    return s
}

func newUser() *User {             // *User escapes — returned pointer
    return &User{Name: "ada"}
}
  • Returning a pointer to a local → escapes to heap.
  • Storing a value in an interface usually escapes (boxing).
  • Allocating slices/maps with size hints (make([]T, 0, n)) avoids growth churn.

Garbage collector

  • Concurrent, tri-color mark-sweep with write barriers. Non-generational, non-compacting.
  • GOGC controls heap-growth target (default 100% = double live heap before next GC). Lower = more frequent, less RAM. GOMEMLIMIT (1.19+) sets a soft cap.
  • STW pauses are typically <1ms; tail latency comes from assist work, not pauses.

unsafe.Pointer bypasses the type system. Allowed conversions are listed in the spec — anything else is undefined behavior. Avoid except for FFI and very-hot paths.

Reflection

func walk(v any) {
    rv := reflect.ValueOf(v)
    if rv.Kind() == reflect.Pointer { rv = rv.Elem() }
    for i := 0; i < rv.NumField(); i++ {
        f := rv.Type().Field(i)
        fmt.Println(f.Name, f.Tag.Get("json"), rv.Field(i).Interface())
    }
}
  • Reflection is slow and loses type safety. Use it for serializers, ORMs, generic frameworks — not for business logic.
  • Prefer struct tags + code generation when possible.

// Common helpers (avoid reinventing these — use golang.org/x/exp/slices, maps)
func Map[A, B any](xs []A, f func(A) B) []B {
    out := make([]B, len(xs))
    for i, x := range xs { out[i] = f(x) }
    return out
}

Profiling — the muscle memory

SymptomReach for
"It's slow"CPU profile (-cpuprofile or /debug/pprof/profile) → flame graph
"Memory keeps growing"Heap profile (-memprofile, alloc_objects view) + runtime.MemStats over time
"Tail latency spikes"runtime/trace + execution tracer; look for GC assist, blocking syscalls, scheduler latency
"Goroutine count climbs"Goroutine profile (/debug/pprof/goroutine?debug=2) → leaks
"Lock contention"runtime.SetMutexProfileFraction + /debug/pprof/mutex
"Where do all the bytes come from?"Heap alloc_space view + benchstat for benchmark deltas

Anti-patterns — what NOT to say

  • "I'll just disable the GC." GOGC=off is a profiling tool, not a deployment strategy.
  • "unsafe.Pointer for speed." 99% of the time you're wrong, and the 1% should be measured against safe code.
  • "Reflection-based generic helpers" — Go 1.18 has actual generics. Use them.
  • "sync.Pool as a long-lived cache." Pooled values can be cleared at any GC cycle.
  • "Optimize first, measure later." A 10x improvement in a 1% function is a 0.1% improvement to the program.
  • Spawning a goroutine to "speed things up" without measurement — context switching and synchronization can dominate.

Senior interviewer follow-ups

ProbeWhat they want to hear
"What does GOGC=200 mean?"Trigger GC when heap doubles past previous live heap × 2 (= 200%). Less GC, more RAM.
"What's GOMEMLIMIT for?"Soft cap on total Go memory (1.19+); GC works harder as you approach it. Best paired with a small GOGC.
"How do generics differ from C++ templates?"Go uses GC shape stenciling: one specialization per memory shape, with itab-style dispatch for method calls. Less code-bloat than C++; some virtual-call cost.
"What does the race detector instrument?"Memory accesses get vector-clock tags; concurrent unsynchronized access is flagged. Sound (no false positives), incomplete (only what runs).
"How do you reduce allocations in a hot path?"Pre-size slices/maps, strings.Builder, sync.Pool for reusable buffers, value receivers for small structs, avoid interface boxing.
"What's an itab?"The runtime descriptor that maps an interface's method set onto a concrete type's methods. Cached per (interface, concrete type) pair.
"How is panic propagated?"Unwinds the stack running deferred functions; if a defer calls recover() it stops; otherwise it kills the process (or just the goroutine — but a panicking goroutine that isn't recovered crashes the whole program).

Senior topics & gotchas

  • Init order: package-level vars, then init() functions, in import-dependency order. Keep init free of side-effects beyond plumbing.
  • Finalizers exist (runtime.SetFinalizer) — almost never the right tool. Use explicit Close()+defer.
  • sync.Pool reduces allocations for short-lived buffers (e.g. bytes.Buffer per request) — but pooled objects can be dropped at any GC.
  • Function inlining limits: small leaf functions get inlined; defer used to disable inlining (no longer in modern Go).
  • Cgo trades portability and stack semantics; each cgo call costs ~200ns and locks an OS thread. Avoid in hot paths.

  • I know when to reach for atomics, sync.Pool, and when not to.
  • I can write a benchmark, capture a CPU/heap profile, and read flame graphs.
  • I know what GOGC and GOMEMLIMIT do and when each helps.
  • I understand the Go memory model and the role of synchronization events.
  • I can use the runtime/trace tool to find scheduler/GC tail latency.
  • I know when generics earn their keep vs interfaces.
  • I can find a goroutine leak using pprof goroutine.
  • I can explain the cgo cost and avoid it in hot paths.
