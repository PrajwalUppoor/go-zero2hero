Concurrency

Goroutines, channels, sync primitives, context — and the patterns interviewers love: pipelines, fan-in/out, worker pools, rate limiting, singleflight, graceful shutdown.

Mental model

  • Goroutine = a function multiplexed onto OS threads by the runtime. Starts at ~2KB stack and grows; an idle Go process can comfortably hold hundreds of thousands of them.
  • GMP scheduler — Goroutines run on Machines (OS threads) bound to Processors (logical CPUs). GOMAXPROCS sets P-count. A goroutine yields on channel ops, syscalls, function calls (preemption since 1.14), and runtime.Gosched().
  • Two flavors of synchronization:
    1. Channels — for transferring ownership of data and signalling between goroutines.
    2. Mutexes / atomics — for protecting state that stays in place.
    "Don't communicate by sharing memory; share memory by communicating" — but a sync.Mutex is often the cleaner answer. Pick the simpler tool.
  • Concurrency ≠ parallelism. Go gives you concurrency (independent things in progress); the runtime adds parallelism on multi-core hardware.

Channels — the cheat-sheet

ch := make(chan int)        // unbuffered = rendezvous (sender blocks until receiver)
ch := make(chan int, 16)    // buffered (sender blocks only when full)

ch <- 1                     // send  (blocks per rules above)
v, ok := <-ch               // receive; ok==false if channel is closed AND drained
close(ch)                   // ONLY the sender(s) close; closing twice panics
for v := range ch { ... }   // exits when channel is closed and drained

select {
case v := <-in:                 handle(v)
case out <- next:               next = produce()
case <-time.After(time.Second): timeout()
case <-ctx.Done():              return ctx.Err()
default:                        // non-blocking
}
Closing a channel signals "no more values"; it is not a destructor. Send on closed = panic. Receive on closed = zero value, ok=false. Multiple senders? Use a separate done channel (or context) instead of closing the data channel.

Buffered vs unbuffered — when to pick which

  • Unbuffered: synchronous handoff. Use for tight coordination ("I'll continue once you've got it").
  • Buffered, size=N: decouples producer from consumer up to N items. Useful for batching / smoothing bursty work. Don't use to "fix" deadlocks — that just delays them.
  • Buffered, size=1: a tiny mailbox / one-shot signal.

Pattern 1 — Worker pool

Use when work items are independent and the bottleneck is CPU/IO of a single worker. Bounding fan-out is critical: unbounded go f() in a hot loop is the #1 way Go services fall over.

func worker(id int, jobs <-chan Job, results chan<- Result, wg *sync.WaitGroup) {
    defer wg.Done()
    for j := range jobs {           // exits when jobs is closed
        results <- process(j)
    }
}

func RunPool(jobs []Job, n int) []Result {
    in  := make(chan Job, n)
    out := make(chan Result, n)
    var wg sync.WaitGroup
    for i := 0; i < n; i++ {
        wg.Add(1)
        go worker(i, in, out, &wg)
    }
    go func() {                     // feed
        for _, j := range jobs { in <- j }
        close(in)
    }()
    go func() { wg.Wait(); close(out) }()  // close out only after all workers done

    var results []Result
    for r := range out { results = append(results, r) }
    return results
}

Complexity: wall-time ≈ total_work / n; memory ≈ O(n + buffer).

Worked trace — 3 jobs, 2 workers

  1. Pool starts: w0, w1 both block on <-jobs.
  2. Feeder sends J1 → w0 receives; sends J2 → w1 receives; sends J3 → blocks until a worker frees.
  3. w0 finishes J1, sends R1 to out, blocks on next job. Feeder unblocks, sends J3 → w0 picks up.
  4. Feeder closes in. Workers each see closed channel after their current job, return.
  5. wg.Wait() returns, the goroutine closes out, the main range loop exits.

Pattern 2 — Pipeline (stages connected by channels)

Each stage is a goroutine that receives from upstream, transforms, and sends downstream. Pair every send/receive with a select { case ... <-ctx.Done(): return } so cancellation propagates and goroutines can't leak.

func gen(ctx context.Context, ns ...int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for _, n := range ns {
            select {
            case out <- n:
            case <-ctx.Done(): return
            }
        }
    }()
    return out
}

func sq(ctx context.Context, in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {
            select {
            case out <- n*n:
            case <-ctx.Done(): return
            }
        }
    }()
    return out
}

// Compose: sq(ctx, sq(ctx, gen(ctx, 1,2,3,4)))

Pattern 3 — Fan-in / fan-out with errgroup

golang.org/x/sync/errgroup is the idiomatic way to launch a bounded fan-out, propagate the first error, and cancel siblings via the derived context.

import "golang.org/x/sync/errgroup"

func FetchAll(ctx context.Context, urls []string) ([]Page, error) {
    g, ctx := errgroup.WithContext(ctx)
    g.SetLimit(8)                                 // fan-out cap
    pages := make([]Page, len(urls))               // per-index write avoids locks
    for i, u := range urls {
        i, u := i, u                              // pin loop vars (pre Go 1.22)
        g.Go(func() error {
            p, err := fetch(ctx, u)
            if err != nil { return err }          // first error cancels ctx
            pages[i] = p
            return nil
        })
    }
    if err := g.Wait(); err != nil { return nil, err }
    return pages, nil
}

Pattern 4 — Rate limiter (token bucket)

import "golang.org/x/time/rate"

l := rate.NewLimiter(100, 200)              // 100 rps, burst 200
if err := l.Wait(ctx); err != nil { return err }
// or non-blocking: if !l.Allow() { return ErrTooManyRequests }
// or N tokens at once: l.WaitN(ctx, 5)

Anti-patterns: hand-rolling a limiter with time.Tick in a loop (drift + leaks); using a buffered channel of "tokens" without refill (one-shot only).

Pattern 5 — Singleflight (request coalescing)

When N goroutines all ask for the same expensive value (cache miss stampede, deduped RPC), collapse them into one in-flight call.

import "golang.org/x/sync/singleflight"

var g singleflight.Group

func GetUser(ctx context.Context, id string) (*User, error) {
    v, err, _ := g.Do(id, func() (any, error) {
        return fetchUserFromDB(ctx, id)
    })
    if err != nil { return nil, err }
    return v.(*User), nil
}

Use DoChan for select-friendly handling; Forget(key) to evict on negative result so the next caller retries.

Pattern 6 — Graceful shutdown

srv := &http.Server{Addr: ":8080", Handler: mux}

go func() {
    if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
        log.Fatalf("listen: %v", err)
    }
}()

// Trap signals
sig := make(chan os.Signal, 1)
signal.Notify(sig, syscall.SIGINT, syscall.SIGTERM)
<-sig

ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
defer cancel()
if err := srv.Shutdown(ctx); err != nil {
    log.Printf("shutdown: %v", err)
}

Drain order: stop accepting → drain in-flight → close DB/cache pools → flush logs/metrics. Pass the same ctx down so workers can react.

sync primitives — when to reach for what

PrimitiveWhenAnti-pattern
sync.MutexProtect a small critical section.Holding across I/O or long work.
sync.RWMutexMany readers, few writers (read >> write).Mostly-write workloads (slower than Mutex).
sync.OnceLazy singleton init.Re-doing work after error — Once doesn't retry.
sync.WaitGroupWait for N goroutines. Add before go; Done deferred.Calling Add from inside the goroutine — race.
sync.CondRare. Wait for arbitrary predicate.Almost always a channel is cleaner.
sync.MapHeavy concurrent reads with disjoint keys per goroutine.General-purpose map — slower than map+Mutex.
sync.PoolReuse short-lived alloc-heavy objects (buffers, decoders).Caching long-lived state — Pool is GC-cleared anytime.
atomic.*Single-word counters/flags. Use atomic.Int64 (1.19+).Compound state — atomicity per word ≠ per struct.

Context — propagation rules

ctx, cancel := context.WithTimeout(parent, 2*time.Second)
defer cancel()                                  // ALWAYS

req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
resp, err := http.DefaultClient.Do(req)         // honours ctx automatically
  • First arg, named ctx. Never store in a struct (sole exception: http.Server.BaseContext-style frameworks).
  • Always defer cancel() — even on success — to release the timer/goroutine the context spawned.
  • context.Value is for request-scoped infrastructure (trace IDs, auth, logger) — not optional parameters.
  • Cancellation is cooperative. A goroutine that doesn't check ctx.Done() will keep running. Combine with channel sends/recvs in a select.
  • errors.Is(err, context.Canceled) vs context.DeadlineExceeded — distinguish "user gave up" from "we ran out of time".

Common deadlocks & data races

  • Sending on an unbuffered channel with no receiver in the same scheduling group.
  • Forgetting to close a channel that drives a range — receivers block forever.
  • Multiple goroutines closing the same channel → panic: close of closed channel. Use sync.Once or invert ownership.
  • Concurrent map writes → fatal error: concurrent map writes (not a regular panic — kills the process).
  • Returning before wg.Wait(): leaked goroutines that may panic later or hold resources.
  • Loop variable captured in goroutine before Go 1.22 — pin with i := i.
  • Using time.After in a hot select loop — it allocates a timer per iteration. Use time.NewTimer + reset.
  • log.Print from a panicking goroutine while another holds the log mutex — circular wait.

Anti-patterns — what NOT to say

  • "Just throw a go on it" — without bounding fan-out, this is how you OOM in production.
  • "I'll close the channel from the receiver." Closing is the sender's responsibility.
  • "I'll use sync.Map for general thread-safety." It's a niche tool; map + RWMutex is usually faster and simpler.
  • "Sleep until X is ready." Use channels / sync.Cond / context.
  • "I'll buffer the channel to avoid the deadlock." A buffer postpones the bug until production load.
  • "Goroutines are free." They're cheap, not free. Each holds a stack and prevents GC of captured vars.
  • Spawning a goroutine inside a request handler that outlives the request without a context — guaranteed leak under load.

Senior interviewer follow-ups

ProbeWhat they want to hear
"How does select choose when multiple cases are ready?"Pseudo-randomly, to prevent starvation.
"What stops a goroutine you can't see anymore?"Nothing — that's a leak. Only context cancellation, channel signals, or returning from the entry function stop it.
"Mutex or channel for a hot counter?"atomic.Int64. Mutex if you have multi-field invariants.
"How do you implement context.WithCancel?"Embed parent's done channel; allocate a new done chan; cancel() closes it; goroutine watches both.
"Why does the race detector exist if Go is memory-safe?"Memory safety prevents segfaults; data races still produce torn / stale reads. go test -race.
"How does runtime.Gosched() differ from time.Sleep(0)?"Gosched yields P; Sleep parks the goroutine on a timer.
"How do you bound goroutines for incoming HTTP work?"Worker pool with bounded queue (drop / 503 on full); or semaphore channel; or errgroup.SetLimit.
"What's the difference between close(ch) and sending a sentinel?"Close fans out to all receivers and supports range; sentinel is per-receiver.

Idiomatic Go for concurrent code

  • Receive-only / send-only types at API boundaries: func consume(in <-chan T), func produce(out chan<- T). The compiler enforces direction.
  • Return channels you create. Encapsulate goroutine ownership inside the constructor; the caller can only read.
  • Prefer errgroup over hand-rolled WaitGroup+error for fan-out work.
  • Always pair go f() in a function with a clear shutdown story (channel close, ctx cancel, or "process lifetime").
  • Test with -race on every CI run. The race detector is sound (no false positives) but only catches races it observes — write tests that exercise concurrency.
  • Use runtime/trace for "why is this slow?" — it shows goroutine blocking, GC, and scheduler activity.

Senior interview prompts

  • Implement a bounded concurrent task runner with priority queue.
  • Build a request coalescer (singleflight) for identical concurrent calls — and explain when it's wrong (different per-request auth, etc.).
  • Implement context.WithCancel from primitives.
  • Design a goroutine-safe LRU with O(1) get/put and TTL.
  • Implement a fan-out broadcast: one publisher, N subscribers, slow subscribers shouldn't block fast ones.
  • Implement a fixed-window and sliding-window rate limiter without external libs.
  • Build a worker pool that supports dynamic resize without dropping in-flight work.
  • Explain how cancellation propagates from ctx down to a TCP Read.

Self-check

  • I can write a worker pool, pipeline, errgroup fan-out, and singleflight from memory.
  • I can explain GMP — what G, M, P are, and what triggers a goroutine yield.
  • I know which of mutex / RWMutex / atomic to pick for a hot counter (and why).
  • I always pair WithCancel/WithTimeout with defer cancel().
  • I understand send-on-closed = panic, recv-on-closed = zero value + ok=false.
  • I know how to bound fan-out (errgroup.SetLimit, semaphore channel, worker pool).
  • I can implement graceful shutdown for an HTTP server (signal → ctx → Shutdown).
  • I know go test -race exists and run it in CI.
  • I can sketch context.WithCancel from primitives (chan + once).
  • I can identify the 3 most common goroutine leaks: missing ctx-check, unclosed channel range, blocking send to no-receiver.
  • I can use runtime/pprof goroutine profile to find a leak in production.
