Binary Search

If you can phrase the question as “what is the smallest x such that P(x) is true?”, you can binary-search the answer space — even when the input array is unsorted.

Mental model

  • Binary search isn't about sorted arrays — it's about monotonic predicates. If P(x) is true for some range and false for the complement (or vice versa), you can binary-search the boundary.
  • Three flavors:
    1. Index in a sorted array (sort.Search, slices.BinarySearch).
    2. Predicate-driven (search the answer) — pick a feasibility function and binary-search the range of possible answers.
    3. Modified arrays (rotated, mountain, peak) — search with local monotonicity.
  • Always use the half-open form [lo, hi) and m = lo + (hi-lo)/2 to avoid overflow and off-by-ones.
  • If you can't make the loop terminate, you have the wrong invariant. Pin down what's true at every iteration before debugging.

The two canonical templates

// 1) Find lower bound (smallest i with a[i] >= target). Half-open [lo, hi).
func lowerBound(a []int, target int) int {
    lo, hi := 0, len(a)
    for lo < hi {
        m := lo + (hi-lo)/2
        if a[m] < target { lo = m + 1 } else { hi = m }
    }
    return lo
}

// 2) Predicate-based: smallest x in [lo, hi] with ok(x).
func searchAnswer(lo, hi int, ok func(int) bool) int {
    for lo < hi {
        m := lo + (hi-lo)/2
        if ok(m) { hi = m } else { lo = m + 1 }
    }
    return lo
}
Compute mid as lo + (hi-lo)/2 to avoid overflow. Use the half-open form [lo, hi) to keep updates symmetric.

Stdlib: sort.Search

// Returns smallest i in [0, n) where f(i) is true; n if none.
i := sort.Search(len(a), func(i int) bool { return a[i] >= target })

Search a rotated sorted array

func search(a []int, t int) int {
    lo, hi := 0, len(a)-1
    for lo <= hi {
        m := lo + (hi-lo)/2
        if a[m] == t { return m }
        if a[lo] <= a[m] { // left half sorted
            if a[lo] <= t && t < a[m] { hi = m - 1 } else { lo = m + 1 }
        } else {           // right half sorted
            if a[m] < t && t <= a[hi] { lo = m + 1 } else { hi = m - 1 }
        }
    }
    return -1
}

Binary search on the answer

// Capacity to ship within D days.
func shipWithinDays(weights []int, D int) int {
    lo, hi := 0, 0
    for _, w := range weights { if w > lo { lo = w }; hi += w }
    ok := func(cap int) bool {
        days, cur := 1, 0
        for _, w := range weights {
            if cur+w > cap { days++; cur = 0 }
            cur += w
        }
        return days <= D
    }
    for lo < hi {
        m := lo + (hi-lo)/2
        if ok(m) { hi = m } else { lo = m + 1 }
    }
    return lo
}

Anti-patterns — what NOT to say

  • Writing m = (lo + hi) / 2 — overflow on huge inputs. Use lo + (hi-lo)/2.
  • Two different update rules in one loop: lo = m AND hi = m — infinite loop.
  • Searching the array when the problem is "find the minimum X such that we can do Y" — that's an answer-search, not an array-search.
  • Hand-rolling lower/upper bound when sort.Search / slices.BinarySearchFunc exist.
  • Reading a[m] without bounds-checking after manipulating hi.

Senior interviewer follow-ups

ProbeWhat they want to hear
"How do you decide between [lo, hi) and [lo, hi]?"Pick one and stay consistent in a problem. Half-open avoids most off-by-ones.
"How do you binary-search on a function f(x) instead of an array?"Find the monotonic predicate; bisect on the answer-range. Worked example: Koko, Capacity to Ship, Allocate Books.
"What if there are duplicates in a rotated array?"Worst case O(n) — when a[lo] == a[mid] == a[hi], you can only safely shrink one side by 1.
"Why is binary search not monotonic in floating point?"Termination must use an epsilon or a fixed iteration count, not equality.
"Find first bad version with min API calls?"Classic sort.Search — return the first index where the predicate flips.
"Lower bound vs upper bound — same code?"Same binary search; predicate flips: a[m] >= target for lower, a[m] > target for upper.

Pitfalls

  • Infinite loops when boundary updates aren't tightened (e.g. lo = m with even mid).
  • Forgetting that sort.Search returns n when no element satisfies — always check.
  • Searching duplicates: prefer lower bound + upper bound rather than a found-or-not loop.
  • Computing mid as (lo+hi)/2 with int overflow on edge sizes.
  • Using == on floats — always epsilon or iteration cap.

Self-check

  • I can write lower-bound and upper-bound from memory.
  • I can binary-search on the answer with a feasibility predicate.
  • I can search a rotated sorted array (no duplicates) in O(log n).
  • I can find the minimum in a rotated sorted array.
  • I can solve Koko Eating Bananas / Capacity to Ship / Split Array Largest Sum.
  • I know the partition trick for Median of Two Sorted Arrays.
  • I never write (lo+hi)/2 — always lo + (hi-lo)/2.
  • I can use sort.Search with a custom predicate.
  • I know how duplicates degrade rotated search to O(n).
  • I can binary-search a float answer with epsilon termination.
